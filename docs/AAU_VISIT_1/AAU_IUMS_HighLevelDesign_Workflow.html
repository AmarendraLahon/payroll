Workflow 1 — Master-Driven Configuration

Nothing is hard-coded. Settings are built by combining master entries — exactly as AAU works today.

Each is a MASTER entry: ┌─ DDO ──────────── (mapped to departments) ├─ Designation ──── (mapped to Payscale) ├─ Grade ├─ Fund Type ────── (mapped to Head & Sector) ├─ Budget Head ├─ Discipline └─ Nature Type │ combine ▼ ┌──────────────────────────────────┐ │ SETTING (a valid combination) │ └──────────────────────────────────┘ │ applies ▼ Payscale ▶ Formula ▶ Head ▶ Deduction% │ ▼ auto-populated, dynamic PAY SLIP (credits + deductions)
  • Designation → Payscale mapping in masters (designation-wise structure — to confirm).
  • Calculation formulas are master entries, mapped to a head; deductions apply across employees.
  • Deduction breakup % maintained in master — LIC, GIS, SSS, IT, loans.
  • Fund Type ↔ Head ↔ Sector and Designation ↔ Fund Type mappings drive which fund pays which salary component.
  • Changing a formula re-drives increments & corrections without code changes.
Effective-dated masters so a past pay run re-computes with the rules that applied then — vital for arrears & audit.

Workflow 4 — Fund Control (FOC)

"Fixation of Ceiling" — multiple fund sources, each spent only on its sanctioned purpose.

Govt yearly budget ──┐ ICAR / scheme grants ─┤ ┌─────────────────────────────┐ Internal receipts ────┼── received ────▶│ FOC — Fixation of Ceiling │ Endowments / other ───┘ │ • fund type ↔ head ↔ sector │ │ • designation ↔ fund mapping│ │ • dispersal breakup as per │ │ requirement on receipt │ └──────────────┬──────────────┘ │ ceiling gates ┌──────────────────────────┼──────────────────────────┐ ▼ ▼ ▼ Salary component Store / Purchase Other expenditure drawn from its fund drawn from its fund within its ceiling │ ▼ Monthly salary bill ▶ Treasury
Gap surfaced in study: tracing the actual source of a fund is currently weak (though managed seamlessly). The system makes fund → head → spend traceable end to end.

Workflow 5 — Bill & Disbursement Chain

The exact approval chain observed in the study — digitised, tracked & auditable.

Payroll BILL TRACKING (each hop logged, with status & timestamp) Funds out Bill + PDF ──▶ Accounts Dept ──▶ generates ──▶ DDO ──▶ Budget ──▶ funds ──▶ Cashier ──▶ releases (upload) receives bill sanctions checks to verifies funds ceiling Accounts │ │ └──────────────── status visible to originator ─────────┘ at every stage (full lifecycle view)
  • Mirrors today's flow: Accounts → DDO → Budget → funds to Accounts → Cashier releases.
  • Bill Tracking gives a single live status; no more chasing files between desks.
  • Each hop checks against FOC ceiling & budget availability before passing on.

Workflow 6 — EFMS (E-File Movement)

The access-controlled file rail that carries every administrative action across the system.

Creator opens file │ full lifecycle view ▼ ┌───────────────────────────┐ │ Received ▶ Accessed ▶ │ │ Forwarded ▶ Processed │ every │ (complete movement log) │ action └─────────────┬─────────────┘ logged │ on processing ▼ Notification ▶ Reporting Officer │ linked by PAN + email across systems
  • Access-based — 3 levels of file access.
  • 6 types of users with distinct rights.
  • Full lifecycle view for the file creator.
  • Complete log: received, accessed, forwarded, processed.
  • On processing → notification to reporting officer.
  • PAN & email are the communication link between systems.
EFMS is the common workflow + audit backbone reused by Bill Tracking, Establishment, Leave, Pension & more.

One master (HRMS), one money convergence point (Finance & Accounts), one file rail (EFMS), one identity (PAN).

Identity & Integration Principles

  • PAN is the universal key — links employee across HRMS, Payroll, EFMS & external APIs.
  • Email + PAN = the cross-system communication link (as in EFMS today).
  • JSON faculty data (qualifications, employment, research) is consumed as-is — no re-keying.
  • 8 appraisal formats integrated into promotion workflow.
Integrate, don't replace. The well-used HRMS and existing logins stay. New modules attach through clean, versioned interfaces and read the existing masters.
Reconciliation built in — payroll-to-bill and fund-to-spend totals must agree, giving audit evidence by design.

Roles, Access & Control

  • 6 user types & 3 access levels (from EFMS) generalised into system-wide RBAC.
  • Separation of duties — maker submits, checker approves on pay, fund & master changes.
  • Centre / department scope — users see only their domain.
  • Reporting-officer notifications on file processing.
Action ─▶ Maker ─▶ Checker ─▶ effect │ │ ▼ ▼ every change written to append-only audit log (who · what · when · why · which order/sanction)
Government-grade auditability: append-only logs & effective-dated rules make every past computation reproducible for AG/CAG audit & RTI.

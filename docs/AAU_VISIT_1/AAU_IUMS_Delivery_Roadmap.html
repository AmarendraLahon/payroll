Prepared by: Pallab Handique  |  For: BSPL Management (cost-estimation input)  |  Version: 0.1 (Draft)  |  Date: 11 June 2026  |  Baseline: Visit Closure Document v1.0 + HLD (8–9 June 2026 study)

How to read this document. This is an effort roadmap, not a price. It sizes every module in person-days across four work types — Req (requirement collection), Des (design), Dev (development), Tst (test/QA) — and orders them by delivery phase and dependency. Management applies day-rates, overheads, margin and contingency on top to produce the formal estimate.
Confidence basis. Numbers are indicative, pre-BRD. They assume the working assumptions in the Closure Document hold. Items flagged Low confidence depend directly on the open points in Closure §6 (fund-source flow, module reality, formula governance, scope boundaries) and on securing the incumbent vendor's legacy code (Closure §7.2). Resolving those can move effort up or down materially.

1. Roadmap Approach & Legend

  • Unit: one person-day = one developer/analyst working day. Effort is build effort only — it excludes day-rate, leave, ramp-up and management margin (management adds those).
  • Work types per module: Req = requirement collection & analysis · Des = solution/data-model design · Dev = development · Tst = test, QA & defect fix.
  • Granularity: each module is broken into sub-features; sub-feature lines carry the Dev estimate; Req/Des/Tst are summed at module level.
  • Sequencing: phases follow the HLD dependency order. Foundation (Phase 0) gates everything; masters & EFMS must exist before payroll/finance workflows run.
Build state: REUSE existing AAU system — integrate, light effort ENHANCE exists but extended/wrapped BUILD new module Confidence: High · Med · Low

9. Risks, Assumptions & Exclusions (effort-affecting)

Risks that move effort

  • Legacy logic lives in incumbent code — handover delay blocks Payroll/Finance/FOC/Pension req-collection (Closure §7.2).
  • Payscale model unresolved — Closure says pay-grade-wise, HLD says designation-wise (to confirm). Reshapes Masters + Payroll.
  • Module reality unknown — paper-only modules may add or remove scope until usage-status sheet received.
  • Fund-source tracing is the stated study gap — FOC effort can swing on actual flow (Closure §6.1).
  • External API maturity unknown — treasury/PFMS/NPS integration depth variable.
  • Scope boundaries open — IT (deduction vs full statutory), loans (lifecycle vs deduction), pension (integrated vs handoff), EFMS-vs-salary-bill engine count (Closure §6.4).

Assumptions (per Closure §5)

  • HRMS data reasonably complete and usable as primary source; PAN unique per employee.
  • AAU provides data, samples, SMEs and decision authority on time.
  • Salary/fund/formula logic made available for reverse-engineering.

Excluded from this effort (unless later agreed)

  • TA/DA inside salary engine (separate billing — see Claims).
  • Offline Store/Inventory in initial phase unless explicitly requested.
  • Modules confirmed unused — not rebuilt by default.
  • Hardware, network/connectivity, third-party licences.
